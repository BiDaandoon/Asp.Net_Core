##### **کپسوله سازی در سی شارپ چیست؟**

کپسوله‌سازی یکی از اصول اساسی برنامه‌نویسی شی‌گرا (OOP) است. در سی‌شارپ، مانند سایر زبان‌های OOP، کپسوله‌سازی به دسته‌بندی داده‌ها و رفتارهای مرتبط در یک واحد (معمولاً یک کلاس) و محدود کردن دسترسی به برخی از اجزای شیء اشاره دارد. این کار به محافظت از یکپارچگی داده‌ها و نمایش تنها موارد ضروری به دنیای خارج کمک می‌کند.

##### **مثال بلادرنگ از اصل کپسوله‌سازی در سی‌شارپ: سیستم مدیریت حساب بانکی**

یک مثال بلادرنگ از کپسوله‌سازی، سیستم مدیریت حساب بانکی است. بیایید سناریویی را در نظر بگیریم که در آن یک مشتری بانک می‌تواند از حساب خود پول واریز یا برداشت کند. با این حال، قوانین و اعتبارسنجی‌های خاصی اعمال می‌شود، مانند اطمینان از اینکه موجودی از حداقل حد مجاز کمتر نشود یا مشتری نتواند بیشتر از موجودی خود در حساب برداشت کند. بیایید ببینیم چگونه می‌توانیم این مثال را با استفاده از اصل کپسوله‌سازی در C# پیاده‌سازی کنیم:

```csharp
using System;

namespace EncapsulationPrincipleCSharp
{
    public class BankAccount
    {
        // This private field is encapsulated and can't be directly accessed from outside the class.
        private decimal balance; 

        public decimal Balance
        {
            // Only provides a way to read the balance but not modify it directly.
            get { return balance; } 
        }

        public BankAccount(decimal initialBalance)
        {
            if (initialBalance < 0)
            {
                throw new ArgumentException("Initial balance cannot be negative.");
            }

            balance = initialBalance;
        }

        public void Deposit(decimal amount)
        {
            if (amount <= 0)
            {
                throw new ArgumentException("Deposit amount should be positive.");
            }

            balance += amount;
        }

        public void Withdraw(decimal amount)
        {
            if (amount <= 0)
            {
                throw new ArgumentException("Withdrawal amount should be positive.");
            }

            if (balance - amount < 0)
            {
                throw new InvalidOperationException("Insufficient funds.");
            }

            balance -= amount;
        }
    }

    //Testing Encapsulation Principle
    public class Program
    {
        static void Main(string[] args)
        {
            // Starts with a balance of 500
            BankAccount myAccount = new BankAccount(500);

            // Balance becomes 700
            myAccount.Deposit(200); 
            Console.WriteLine(myAccount.Balance); // Outputs: 700

            // Balance becomes 600
            myAccount.Withdraw(100); 
            Console.WriteLine(myAccount.Balance); // Outputs: 600

            // myAccount.balance = -1000;  // This would be an error, as the balance field is private and inaccessible directly.
            
            Console.Read();
        }
    }
}
```

**در این مثال:**

- کلاس BankAccount فیلد موجودی را کپسوله‌سازی می‌کند، به این معنی که نمی‌توان مستقیماً از خارج از کلاس به آن دسترسی داشت یا آن را تغییر داد. در عوض، عملیاتی مانند واریز یا برداشت توسط متدهای (Deposit و Withdraw) کنترل می‌شوند.
- ویژگی Balance فقط یک geter ارائه می‌دهد و تضمین می‌کند که موجودی قابل مشاهده است اما نمی‌توان آن را مستقیماً تغییر داد.
- روش‌های واریز و برداشت، قوانین تجاری سیستم بانکی ما را در بر می‌گیرد، مانند اطمینان از اینکه شما نمی‌توانید موجودی اولیه منفی داشته باشید، نمی‌توانید مبالغ منفی واریز کنید و نمی‌توانید بیش از آنچه دارید برداشت کنید.

این رویکرد، یکپارچگی داده‌های حساب را حفظ می‌کند و تضمین می‌کند که نیروهای خارجی نمی‌توانند از آن سوءاستفاده یا سوءاستفاده کنند.

##### **مثال کپسوله‌سازی بلادرنگ در سی‌شارپ: دستگاه قهوه‌ساز**

بیایید یک مثال بلادرنگ دیگر از دستگاه قهوه‌ساز را در نظر بگیریم. یک دستگاه قهوه‌ساز مکانیسم‌ها و عملیات داخلی (مانند آسیاب کردن دانه‌های قهوه، گرم کردن آب و غیره) دارد که کاربر نباید نگران آنها باشد. کاربر نوع قهوه مورد نظر خود را انتخاب می‌کند و دستگاه محصول نهایی را تحویل می‌دهد. بیایید ببینیم چگونه می‌توانیم این مثال را با استفاده از اصل کپسوله‌سازی در C# پیاده‌سازی کنیم:

```csharp
using System;

namespace EncapsulationPrincipleCSharp
{
    public class CoffeeMachine
    {
        private int waterAmount; // in milliliters
        private int beansAmount; // in grams
        private bool isHeated;

        public CoffeeMachine(int water, int beans)
        {
            waterAmount = water;
            beansAmount = beans;
            isHeated = false;
        }

        private void HeatWater()
        {
            if (!isHeated)
            {
                Console.WriteLine("Heating water...");
                isHeated = true;
            }
        }

        private void GrindBeans(int amount)
        {
            if (beansAmount < amount)
            {
                throw new InvalidOperationException("Not enough coffee beans!");
            }
            Console.WriteLine("Grinding coffee beans...");
            beansAmount -= amount;
        }

        // This is the method exposed to the user.
        public void MakeEspresso()
        {
            HeatWater();
            GrindBeans(20); // let's say we need 20 grams of beans for an espresso
            Console.WriteLine("Making Espresso...");
        }

        // Another method exposed to the user.
        public void MakeLatte()
        {
            HeatWater();
            GrindBeans(25); // we need 25 grams of beans for a latte
            Console.WriteLine("Making Latte...");
        }

        public int BeansLeft()
        {
            return beansAmount;
        }

        public int WaterLeft()
        {
            return waterAmount;
        }
    }
    
    //Testing Encapsulation Principle
    public class Program
    {
        static void Main(string[] args)
        {
            CoffeeMachine myMachine = new CoffeeMachine(1000, 100);  // Initialize with 1000 ml of water and 100 grams of beans

            myMachine.MakeEspresso();  // Outputs: Heating water... Grinding coffee beans... Making Espresso...

            Console.WriteLine($"Beans left: {myMachine.BeansLeft()} grams");  // Outputs: Beans left: 80 grams

            Console.Read();
        }
    }
}
```

**در مثال بالا:**

- کلاس CoffeeMachine عملکرد داخلی دستگاه را کپسوله‌سازی می‌کند. متدهایی مانند HeatWater و GrindBeans خصوصی هستند و خارج از کلاس قابل دسترسی نیستند.
- فقط متدهایی که برای استفاده عمومی در نظر گرفته شده‌اند، مانند MakeEspresso و MakeLatte، در معرض دید قرار می‌گیرند. این امر تضمین می‌کند که کاربر به شیوه‌ای کنترل‌شده با دستگاه تعامل داشته باشد.
- وضعیت داخلی دستگاه قهوه (مقدار آب و دانه‌های قهوه، چه آب گرم شده باشد چه نشده باشد) از دستکاری‌های خارجی محافظت می‌شود و یکپارچگی آن تضمین می‌گردد.

این یک نمایش ساده از یک دستگاه قهوه است، اما مفهوم کپسوله‌سازی را نشان می‌دهد: پنهان کردن پیچیدگی داخلی و نمایش فقط آنچه برای کاربر ضروری است. وقتی کد بالا را اجرا می‌کنید، خروجی زیر را دریافت خواهید کرد:

##### **مثال کپسوله‌سازی بلادرنگ در سی‌شارپ: سرعت‌سنج خودرو**

یک مثال واقعی دیگر را در نظر بگیرید: سرعت‌سنج خودرو و اجزای مرتبط با آن. سرعت‌سنج خودرو، سرعت فعلی را به راننده نمایش می‌دهد. در داخل، این سرعت توسط چندین عامل و محاسبات مربوط به چرخش چرخ‌ها و نسبت دنده‌ها تعیین می‌شود، اما راننده فقط باید مقدار نهایی سرعت را ببیند. آنها نیازی به دانستن (و اغلب نمی‌خواهند بدانند) محاسبات و مکانیسم‌های پیچیده‌ی مربوطه ندارند. بیایید ببینیم چگونه می‌توانیم این مثال را با استفاده از اصل کپسوله‌سازی در C# پیاده‌سازی کنیم:

```csharp
using System;

namespace EncapsulationPrincipleCSharp
{
    public class Car
    {
        private int wheelRotations;
        private double gearRatio = 4.2;  // Simplified representation of gear ratio
        private const double WHEEL_CIRCUMFERENCE = 2.0; // Assuming each wheel rotation covers 2 meters

        // Constructor to initialize the car
        public Car()
        {
            wheelRotations = 0;
        }

        // This method simulates the car moving and the wheel rotating
        private void Move()
        {
            // For simplicity, we're assuming each call to Move represents one wheel rotation.
            wheelRotations++;
        }

        private double CalculateSpeed()
        {
            // Calculates speed based on wheel rotations and gear ratio
            double distanceCovered = wheelRotations * WHEEL_CIRCUMFERENCE;
            double time = 1.0; // Assume 1 second for simplicity
            return (distanceCovered * gearRatio) / time;
        }

        // Public method to drive the car
        public void Drive()
        {
            Move();
            Console.WriteLine($"Current speed: {CalculateSpeed()} m/s");
        }

        // Let's say there's a mechanism to reset the speedometer
        public void ResetSpeedometer()
        {
            wheelRotations = 0;
        }
    }
    
    //Testing Encapsulation Principle
    public class Program
    {
        static void Main(string[] args)
        {
            Car myCar = new Car();
            myCar.Drive();  // Outputs: Current speed: 8.4 m/s
            myCar.Drive();  // Outputs: Current speed: 16.8 m/s

            Console.Read();
        }
    }
}
```

**در مثال بالا:**

- مکانیزم داخلی ماشین (مثلاً متد Move و متغیر wheelRotations) از دید راننده پنهان است.
- درایور فقط با متدهای Drive و ResetSpeedometer که برای استفاده عمومی در دسترس هستند، تعامل دارد.
- متد CalculateSpeed، اگرچه داخلی است، اما منطق مربوط به تعیین سرعت فعلی بر اساس چرخش چرخ‌ها و نسبت دنده را مدیریت می‌کند.

در اینجا، پیچیدگی محاسبه سرعت و مدیریت حالت داخلی مربوط به چرخش چرخ‌ها، درون کلاس Car کپسوله‌سازی شده است و راننده فقط نتیجه نهایی سرعت را می‌بیند.

##### **اصل کپسوله‌سازی مثال بلادرنگ در سی‌شارپ: کیف پول دیجیتال**

بیایید یک مثال متفاوت برای درک اصل کپسوله‌سازی در سی‌شارپ، یعنی یک کیف پول دیجیتال، ببینیم. یک کیف پول دیجیتال پول را ذخیره می‌کند و کاربران می‌توانند عملیاتی مانند واریز، برداشت و بررسی موجودی خود را انجام دهند. در داخل، ممکن است بررسی‌های مختلف، رمزگذاری و ثبت تراکنش‌ها وجود داشته باشد، اما این جزئیات از کاربر انتزاع می‌شوند. بیایید ببینیم چگونه می‌توانیم این مثال را با استفاده از اصل کپسوله‌سازی در سی‌شارپ پیاده‌سازی کنیم:

```csharp
using System;
using System.Collections.Generic;

namespace EncapsulationPrincipleCSharp
{
    public class DigitalWallet
    {
        private decimal balance;
        private string walletPassword;
        private List<string> transactionLog = new List<string>();

        public DigitalWallet(string initialPassword)
        {
            balance = 0m;
            walletPassword = initialPassword;
        }

        private void LogTransaction(string transactionDetail)
        {
            // This can be more complex, involving timestamps and other details.
            transactionLog.Add(transactionDetail);
        }

        public void Deposit(decimal amount)
        {
            if (amount <= 0)
            {
                throw new ArgumentException("Deposit amount should be positive.");
            }

            balance += amount;
            LogTransaction($"Deposited {amount}. Current Balance: {balance}");
        }

        public bool Withdraw(decimal amount, string password)
        {
            if (amount <= 0)
            {
                throw new ArgumentException("Withdrawal amount should be positive.");
            }

            if (walletPassword != password)
            {
                LogTransaction($"Failed withdrawal attempt of {amount}. Incorrect password.");
                return false;
            }

            if (balance - amount < 0)
            {
                LogTransaction($"Failed withdrawal attempt of {amount}. Insufficient funds.");
                return false;
            }

            balance -= amount;
            LogTransaction($"Withdrew {amount}. Current Balance: {balance}");
            return true;
        }

        public decimal CheckBalance(string password)
        {
            if (walletPassword == password)
            {
                return balance;
            }
            else
            {
                throw new UnauthorizedAccessException("Incorrect password.");
            }
        }

        // This can be used for auditing purposes.
        public List<string> GetTransactionLog(string password)
        {
            if (walletPassword == password)
            {
                return new List<string>(transactionLog);  // Return a copy to maintain encapsulation.
            }
            else
            {
                throw new UnauthorizedAccessException("Incorrect password.");
            }
        }
    }
    
    //Testing Encapsulation Principle
    public class Program
    {
        static void Main(string[] args)
        {
            DigitalWallet myWallet = new DigitalWallet("securePass123");

            myWallet.Deposit(200m);
            bool isWithdrawn = myWallet.Withdraw(50m, "securePass123"); // Outputs: true

            decimal currentBalance = myWallet.CheckBalance("securePass123"); // Outputs: 150
            
            Console.Read();
        }
    }
}
```

**در این مثال:**

- کلاس DigitalWallet موجودی و گزارش تراکنش‌ها را کپسوله‌سازی می‌کند.
- روش‌های واریز و برداشت با بررسی‌های ایمنی طراحی شده‌اند و گزارش تراکنش‌ها، تمام تراکنش‌ها را ثبت می‌کند.
- تأیید رمز عبور برای اقدامات خاص ضروری است و امنیت را افزایش می‌دهد.
- GetTransactionLog یک کپی از لیست گزارش تراکنش‌ها را برمی‌گرداند تا از تغییرات خارجی جلوگیری کند و بیشتر بر کپسوله‌سازی تأکید دارد.

کاربر با روش‌های عمومی شفاف تعامل می‌کند در حالی که کیف پول منطق داخلی پیچیده، بررسی‌ها و ثبت سوابق را مدیریت می‌کند.

##### **اصل کپسوله‌سازی - مثال بلادرنگ در سی‌شارپ: مدیریت مجوزهای کاربر در یک سیستم نرم‌افزاری**

بیایید کپسوله‌سازی را به طور متفاوتی درک کنیم: مدیریت مجوزهای کاربر در یک سیستم نرم‌افزاری. سیستمی را تصور کنید که در آن کاربران نقش‌های مختلفی مانند "مدیر"، "کاربر" و "مهمان" دارند. هر نقش مجوزهای مختلفی مانند "خواندن"، "نوشتن" و "حذف" دارد. سیستم باید این مجوزها را کپسوله‌سازی و مدیریت کند تا امنیت و دسترسی مناسب تضمین شود. بیایید ببینیم چگونه می‌توانیم این مثال را با استفاده از اصل کپسوله‌سازی در C# پیاده‌سازی کنیم:

```csharp
using System;
using System.Collections.Generic;

namespace EncapsulationPrincipleCSharp
{
    public class User
    {
        private string name;
        private Role userRole;

        public User(string name, Role role)
        {
            this.name = name;
            this.userRole = role;
        }

        public string Name => name;

        public bool HasPermission(PermissionType permission)
        {
            return userRole.HasPermission(permission);
        }
    }

    public enum PermissionType
    {
        Read,
        Write,
        Delete
    }

    public class Role
    {
        private string roleName;
        private List<PermissionType> permissions = new List<PermissionType>();

        public Role(string roleName, List<PermissionType> permissions)
        {
            this.roleName = roleName;
            this.permissions = permissions;
        }

        public bool HasPermission(PermissionType permission)
        {
            return permissions.Contains(permission);
        }
    }

    public class SystemManager
    {
        public void AccessResource(User user, PermissionType permission)
        {
            if (user.HasPermission(permission))
            {
                Console.WriteLine($"{user.Name} has {permission} permission and can access the resource.");
            }
            else
            {
                Console.WriteLine($"{user.Name} does not have {permission} permission and cannot access the resource.");
            }
        }
    }
    
    //Testing Encapsulation Principle
    public class Program
    {
        static void Main(string[] args)
        {
            Role adminRole = new Role("Admin", new List<PermissionType> { PermissionType.Read, PermissionType.Write, PermissionType.Delete });
            Role userRole = new Role("User", new List<PermissionType> { PermissionType.Read, PermissionType.Write });

            User alice = new User("Alice", adminRole);
            User bob = new User("Bob", userRole);

            SystemManager manager = new SystemManager();
            manager.AccessResource(alice, PermissionType.Delete); // Outputs: Alice has Delete permission and can access the resource.
            manager.AccessResource(bob, PermissionType.Delete); // Outputs: Bob does not have Delete permission and cannot access the resource.

            Console.Read();
        }
    }
}
```

**در این مثال:**

- کلاس User نام و نقش مرتبط با آن را کپسوله‌سازی می‌کند. این کلاس نحوه بررسی مجوزها یا جزئیات نقش آنها را نشان نمی‌دهد، اما یک رابط کاربری واضح (HasPermission) برای بررسی مجوزها ارائه می‌دهد.
- کلاس نقش، نام یک نقش و مجوزهای مرتبط با آن را در بر می‌گیرد.
- SystemManager از کپسوله‌سازی در کلاس‌های User و Role برای مدیریت و کنترل دسترسی به منابع استفاده می‌کند.

این طراحی، پیچیدگی بررسی مجوزها را پنهان می‌کند، روشی سازگار و ایمن برای بررسی مجوزها را تضمین می‌کند و رابط کاربری واضحی برای تعامل فراهم می‌کند.

##### **مثال کپسوله‌سازی بلادرنگ در سی‌شارپ: سیستم مدیریت کتاب کتابخانه**

در اینجا یک مثال واقعی دیگر وجود دارد که کپسوله‌سازی را با استفاده از یک سیستم مدیریت کتاب کتابخانه نشان می‌دهد. در این مثال، ما یک کلاس LibraryBook داریم. هر کتاب دارای یک عنوان، یک ISBN و یک وضعیت است که نشان می‌دهد آیا کتاب در حال حاضر امانت داده شده است یا خیر. این کلاس فقط باید اجازه دهد وضعیت با استفاده از روش‌های تعیین‌شده، مانند CheckOut و ReturnBook، تغییر کند تا اطمینان حاصل شود که سیستم وضعیت کتاب را به درستی ردیابی می‌کند. بیایید ببینیم چگونه می‌توانیم این مثال را با استفاده از اصل کپسوله‌سازی در C# پیاده‌سازی کنیم:

```csharp
using System;

namespace EncapsulationPrincipleCSharp
{
    public class LibraryBook
    {
        private string title;
        private string isbn;
        private bool isCheckedOut;

        public LibraryBook(string title, string isbn)
        {
            this.title = title;
            this.isbn = isbn;
            this.isCheckedOut = false;
        }

        public string Title => title;
        public string ISBN => isbn;

        public bool IsCheckedOut => isCheckedOut;

        public void CheckOut()
        {
            if (isCheckedOut)
            {
                Console.WriteLine($"{title} is already checked out.");
            }
            else
            {
                isCheckedOut = true;
                Console.WriteLine($"Successfully checked out {title}.");
            }
        }

        public void ReturnBook()
        {
            if (!isCheckedOut)
            {
                Console.WriteLine($"{title} is not checked out. No need to return.");
            }
            else
            {
                isCheckedOut = false;
                Console.WriteLine($"Successfully returned {title}.");
            }
        }
    }
    
    //Testing Encapsulation Principle
    public class Program
    {
        static void Main(string[] args)
        {
            LibraryBook book = new LibraryBook("Moby Dick", "1234567890");

            book.CheckOut(); // Outputs: Successfully checked out Moby Dick.
            book.CheckOut(); // Outputs: Moby Dick is already checked out.

            book.ReturnBook(); // Outputs: Successfully returned Moby Dick.

            Console.Read();
        }
    }
}
```

**در این مثال:**

- وضعیت داخلی LibraryBook (یعنی اینکه آیا بررسی شده است یا خیر) درون کلاس کپسوله‌سازی شده است. موجودیت‌های خارجی نمی‌توانند مستقیماً این وضعیت را تغییر دهند.
- وضعیت کتاب با استفاده از متدهای خاص (CheckOut و ReturnBook) تغییر می‌کند که شامل منطقی برای اطمینان از انجام صحیح عملیات هستند.
- کاربران کلاس می‌توانند از طریق ویژگی IsCheckedOut ببینند که آیا کتاب خارج شده است یا خیر، اما نمی‌توانند مستقیماً وضعیت را تغییر دهند.

این کپسوله‌سازی تضمین می‌کند که سیستم می‌تواند به طور قابل اعتمادی وضعیت هر کتاب را پیگیری کند و از سوءاستفاده جلوگیری کند.

##### **مزایا و معایب اصل کپسوله‌سازی در سی‌شارپ**

کپسوله‌سازی یکی از چهار اصل اساسی برنامه‌نویسی شی‌گرا (OOP) است. این به دسته‌بندی داده‌ها (ویژگی‌ها) و روش‌ها (توابع) که روی داده‌ها عمل می‌کنند در یک واحد یا کلاس واحد و محدود کردن دسترسی مستقیم به برخی از اجزای شیء اشاره دارد. کپسوله‌سازی یک اصل کلیدی در C# است، مانند بسیاری از زبان‌های شی‌گرای دیگر. در اینجا مزایا و معایب آن آمده است:

###### **مزایای اصل کپسوله‌سازی در سی‌شارپ:**

- **قابلیت نگهداری:** کپسوله‌سازی به سازماندهی کد برای قابلیت نگهداری کمک می‌کند. از آنجایی که عملکردهای داخلی پنهان هستند، تغییرات در منطق یا داده‌ها بر کد خارجی که از شیء استفاده می‌کند تأثیری نمی‌گذارد.
- **انعطاف‌پذیری و توسعه‌پذیری:** شما می‌توانید پیاده‌سازی داخلی یک کلاس را بدون تأثیر بر کلاس‌هایی که از آن استفاده می‌کنند، تغییر دهید.
- **کنترل:** با نمایش تنها بخش‌های ضروری شیء، می‌توانید کنترل بالایی بر عملکرد و داده‌های کلاس داشته باشید. برای مثال، می‌توانید از اعتبارسنجی درون متد set یک کلاس برای اعمال محدودیت‌های خاص استفاده کنید.
- **افزایش امنیت:** کپسوله‌سازی با اجازه دادن به عملیات کنترل‌شده و معتبر روی داده‌ها، از یکپارچگی آنها محافظت می‌کند. این امر از دسترسی غیرمجاز و تغییر تصادفی داده‌ها جلوگیری می‌کند.
- **انتزاع:** این امر جدایی روشنی بین کاری که یک شیء انجام می‌دهد و نحوه دستیابی به آن کار ایجاد می‌کند. این جداسازی به توسعه‌دهندگان اجازه می‌دهد تا با درک رابط‌های بین اشیاء، بدون دانستن سازوکار داخلی، روی سیستم‌های پیچیده کار کنند.
- **کاهش عوارض جانبی:** با محدود کردن دسترسی مستقیم به داده‌های شیء، می‌توانید اطمینان حاصل کنید که تمام عملیات روی آن داده‌ها از طریق رابط‌های خوش‌تعریف انجام می‌شود، که این امر خطاهای پیش‌بینی نشده یا عوارض جانبی را کاهش می‌دهد.

###### **معایب اصل کپسوله‌سازی در سی‌شارپ:**

- **پیچیدگی:** کپسوله‌سازی می‌تواند در طراحی یک سیستم پیچیدگی ایجاد کند، به خصوص اگر بیش از حد یا به طور نامناسب استفاده شود. همه چیز نیازی به کپسوله‌سازی ندارد.
- **سربار:** معرفی getterها و setterها (متدها) برای ویژگی‌ها به جای دسترسی مستقیم به آنها می‌تواند مقدار کمی سربار ایجاد کند. کامپایلرهای مدرن می‌توانند این سربار را بهینه کنند، اما هنوز هم از نظر تئوری یک عیب محسوب می‌شود.
- **زمان توسعه:** در ابتدا، نوشتن کد کپسوله شده می‌تواند کندتر باشد زیرا شما برای همه چیز رابط کاربری ایجاد می‌کنید و دسترسی محدود را تضمین می‌کنید. با این حال، این اغلب در دراز مدت با نگهداری آسان‌تر و خطاهای کمتر، نتیجه می‌دهد.
- **سوءاستفاده احتمالی:** اگر کپسوله‌سازی به درستی استفاده نشود، ممکن است حس امنیت کاذبی ایجاد کند. برای مثال، افشای یک شیء قابل تغییر از طریق یک ویژگی ممکن است تصوری از امنیت ایجاد کند، در حالی که در واقع، وضعیت داخلی شیء همچنان می‌تواند به طور ناخواسته تغییر کند.
- **بده‌بستان‌های دسترسی:** گاهی اوقات، برای کپسوله‌سازی، توسعه‌دهندگان ممکن است اجزای خاصی را خصوصی کنند که ممکن است نیاز به گسترش یا دسترسی به آنها بعداً داشته باشند و نیاز به بازسازی داشته باشند.

##### **چه زمانی از اصل کپسوله‌سازی در سی‌شارپ استفاده کنیم؟**

کپسوله‌سازی یک اصل اساسی در برنامه‌نویسی شی‌گرا است و استفاده‌ی هوشمندانه از آن می‌تواند طراحی، قابلیت نگهداری و استحکام نرم‌افزار شما را تا حد زیادی بهبود بخشد. در اینجا چند سناریو و مورد استفاده‌ی رایج وجود دارد که باید در آن‌ها اصل کپسوله‌سازی را در سی‌شارپ به کار ببرید:

- **هنگام محافظت از یکپارچگی داده‌ها:** اگر لازم باشد وضعیت داخلی یک شیء از مقادیر نامعتبر یا متناقض محافظت شود، کپسوله‌سازی به شما امکان می‌دهد منطق اعتبارسنجی را در تنظیم‌کننده‌ها یا از طریق متدها اعمال کنید.
- **برای پنهان کردن جزئیات پیاده‌سازی:** وقتی می‌خواهید آزادی عمل داشته باشید تا بتوانید عملکردهای داخلی یک کلاس را بدون تأثیرگذاری بر کاربران خارجی کلاس تغییر دهید، کپسوله‌سازی می‌تواند این جزئیات را پشت یک رابط پایدار پنهان کند.
- **هنگام ایجاد کامپوننت‌های قابل استفاده مجدد:** هنگام ساخت کتابخانه‌ها یا کامپوننت‌هایی که قرار است در بخش‌های مختلف یک برنامه یا حتی توسط برنامه‌های دیگر استفاده شوند، کپسوله‌سازی تضمین می‌کند که کاربران این کامپوننت‌ها بدون وابستگی به مکانیسم‌های داخلی آنها، به روش مورد نظر با آنها تعامل داشته باشند.
- **هنگام مدیریت وابستگی‌ها:** کپسوله‌سازی می‌تواند به جداسازی بخش‌های مختلف سیستم کمک کند تا تغییرات در یک جزء به کل برنامه سرایت نکند. این امر به دستیابی به درجه پایینی از اتصال بین اجزای نرم‌افزار کمک می‌کند.
- **برای اجرای الگوهای استفاده:** اگر توالی یا الگوی خاصی وجود دارد که عملیات باید روی یک شیء انجام شود، کپسوله‌سازی می‌تواند به اطمینان از رعایت این الگوها کمک کند.
- **در سناریوهای همزمانی:** هنگام کار با برنامه‌های چند رشته‌ای، کپسوله‌سازی می‌تواند به محافظت از منابع مشترک در برابر مشکلات دسترسی همزمان کمک کند، به عنوان مثال، با کپسوله‌سازی مکانیسم‌های قفل‌گذاری درون شیء.
- **برای دستیابی به انتزاع:** در مواردی که می‌خواهید یک سیستم پیچیده را با رابط کاربری ساده‌تری نمایش دهید، کپسوله‌سازی می‌تواند پیچیدگی‌های اساسی را انتزاع کند.
- **هنگام گروه‌بندی داده‌ها و رفتارهای مرتبط:** کپسوله‌سازی امکان دسته‌بندی داده‌های مرتبط و عملیاتی که روی آنها عمل می‌کنند را فراهم می‌کند و کد را سازمان‌یافته‌تر و فهم آن را آسان‌تر می‌سازد.
- **برای ایجاد انواع سفارشی با رفتار خاص:** به عنوان مثال، اگر در حال ایجاد یک ساختار داده سفارشی یا یک نوع خاص دامنه هستید، کپسوله‌سازی می‌تواند تضمین کند که رفتار و ساختار این نوع همیشه سازگار است.
- **برای بهبود قابلیت تست:** اجزای کپسوله شده به راحتی قابل شبیه‌سازی و تست به صورت جداگانه هستند، زیرا تعاملات آنها از طریق رابط‌های واضح تعریف شده است.

با این حال، یافتن تعادل ضروری است. کپسوله‌سازی بیش از حد می‌تواند منجر به کدی با خوانایی کمتر شود، که در آن هر ویژگی داده و عملیات بدون هیچ مزیت مشخصی پشت accessorها و mutatorها قفل می‌شود. به عنوان یک قاعده کلی، شما باید زمانی کپسوله‌سازی کنید که مزیت مشخصی در مورد یکپارچگی داده‌ها، انتزاع یا قابلیت نگهداری وجود داشته باشد، نه اینکه این کار را برای هر ویژگی یا متد کلاس انجام دهید.
