## **مخرب‌ها در سی شارپ به همراه مثال**

در این مقاله، قصد دارم **به همراه مثال‌هایی در مورد مخرب‌ها در سی‌شارپ** صحبت کنم. لطفاً مقاله قبلی ما را که در آن به همراه مثال‌هایی در مورد **سازنده‌های استاتیک در مقابل سازنده‌های غیر استاتیک در سی‌شارپ** بحث کردیم، مطالعه کنید.

##### **ویرانگرها (Destructor) در سی شارپ چیستند؟**

طبق **[MSDN](https://docs.microsoft.com/en-us/dotnet/csharp/programming-guide/classes-and-structs/finalizers)** ، مخرب‌ها که در C# به آنها Finalizer نیز گفته می‌شود، برای انجام هرگونه پاکسازی نهایی لازم هنگام جمع‌آوری یک نمونه کلاس توسط garbage collector استفاده می‌شوند.

تابع مخرب (Destructor) نیز نوع خاصی از متد است که در یک کلاس وجود دارد، درست مانند تابع سازنده (Constructor)، که نامی مشابه نام کلاس دارد اما با پیشوند **~** شروع می‌شود. تابع سازنده (Constructor) در سی شارپ (C#) به صورت صریح (Explicitly) هنگام ایجاد شیء کلاس فراخوانی می‌شود. از سوی دیگر، تابع مخرب (Destructor) در سی شارپ (C#) به صورت ضمنی (implicitly) هنگام تخریب شیء کلاس فراخوانی می‌شود.

متدهای سازنده و مخرب دقیقاً همنام کلاسی هستند که به آن تعلق دارند. بنابراین، برای تمایز بین این دو، از نماد ~ درست قبل از متد مخرب استفاده می‌شود. سینتکس آن در زیر نشان داده شده است.

![ویرانگرها (Destructor) در سی شارپ چیستند؟](https://dotnettutorials.net/wp-content/uploads/2018/07/word-image-1131-1.png "ویرانگرها (Destructor) در سی شارپ چیستند؟")

**نکته:** مهمترین نکته‌ای که باید در نظر داشته باشید این است که یک متد مخرب نمی‌تواند هیچ پارامتری داشته باشد و همچنین نمی‌تواند با هیچ اصلاح‌کننده‌ای اعمال شود. از آنجایی که مخرب پارامتردهی نشده است، بنابراین نمی‌توانیم مخرب را overload کنیم.

##### **چه زمانی یک متد مخرب (Destructor) در سی شارپ فراخوانی می‌شود؟**

یک متد مخرب (destructor) به طور خودکار توسط garbage collector هنگام از بین رفتن شیء کلاس فراخوانی می‌شود. بنابراین، نکته‌ای که باید به خاطر داشته باشید این است که متدهای مخرب به طور خودکار توسط garbage collector فراخوانی می‌شوند.

##### **چه زمانی شیء یک کلاس در سی شارپ از بین می‌رود؟**

شیء یک کلاس در سی شارپ در هر یک از موارد زیر توسط زباله روب از بین خواهد رفت:

1. در پایان اجرای برنامه، هر شیء مرتبط با برنامه توسط زباله‌روب از بین می‌رود.
2. فراخوانی ضمنی زباله‌روب زمانی در اواسط اجرای برنامه رخ می‌دهد که حافظه پر باشد، بنابراین زباله‌روب اشیاء بلااستفاده برنامه را شناسایی کرده و آنها را از بین می‌برد.
3. فراخوانی صریح garbage collector همچنین می‌تواند در میانه اجرای برنامه توسط برنامه‌نویس با کمک دستور « **Collect()** » انجام شود، به طوری که اگر اشیاء استفاده نشده‌ای مرتبط با برنامه وجود داشته باشد، در میانه اجرای برنامه از بین بروند.

##### **مثال برای درک تابع‌های مخرب در سی شارپ:**

در مثال زیر، ما یک کلاس به نام DestructorDemo با یک سازنده و یک مخرب ایجاد کرده‌ایم. سپس در داخل متد Main، دو نمونه از کلاس DestructorDemo ایجاد کرده‌ایم و سپس مقدار obj1 را null قرار داده‌ایم که واجد شرایط جمع‌آوری زباله است.

```csharp
using System;

namespace DestructorExample
{
    class DestructorDemo
    {
        public DestructorDemo()
        {
            Console.WriteLine("Constructor Object Created");
        }
        ~DestructorDemo()
        {
            string type = GetType().Name;
            Console.WriteLine($"Object {type} is Destroyed");
        }
    }

    class Program
    {
        static void Main(string[] args)
        {
            DestructorDemo obj1 = new DestructorDemo();
            DestructorDemo obj2 = new DestructorDemo();

            //Making obj1 for Garbage Collection
            obj1 = null;
            Console.ReadKey();
        }
    }
}
```

حالا، وقتی کد بالا را اجرا کنید، خروجی زیر را دریافت خواهید کرد.

![مثالی برای درک تابع‌های مخرب در سی شارپ](https://dotnettutorials.net/wp-content/uploads/2018/07/word-image-1131-2.png "مثالی برای درک تابع‌های مخرب در سی شارپ")

می‌بینید که دستورات نوشته شده درون تابع مخرب در خروجی چاپ نمی‌شوند. پس ممکن است فکر کنید که تابع مخرب ما اجرا نمی‌شود، حتی با اینکه مقدار obj2 را null قرار داده‌ایم. این درست نیست. نکته‌ای که باید به خاطر داشته باشید این است که متد تابع مخرب به طور ضمنی توسط garbage collector فراخوانی می‌شود و ما نمی‌توانیم پیش‌بینی کنیم که چه زمانی تابع مخرب فراخوانی می‌شود. و از این رو نمی‌توانید آن دستورات چاپ را در خروجی ببینید. زمانی که شیء بلااستفاده شد، یعنی آماده garbage collection شد، مسئولیت garbage collector است که آن شیء بلااستفاده را از بین ببرد و حافظه را از heap آزاد کند.

اگر بخواهید، می‌توانید در حین اجرای برنامه، فراخوانی صریحی به garbage collector انجام دهید تا شیء استفاده نشده را از بین ببرد. برای انجام این کار، باید **متد GC.Collect** را همانطور که در مثال زیر نشان داده شده است، فراخوانی کنیم.

```csharp
using System;

namespace DestructorExample
{
    class DestructorDemo
    {
        public DestructorDemo()
        {
            Console.WriteLine("Constructor Object Created");
        }
        ~DestructorDemo()
        {
            string type = GetType().Name;
            Console.WriteLine($"Object {type} is Destroyed");
        }
    }

    class Program
    {
        static void Main(string[] args)
        {
            DestructorDemo obj1 = new DestructorDemo();
            DestructorDemo obj2 = new DestructorDemo();

            //Making obj1 for Garbage Collection
            obj1 = null;
            GC.Collect();
            Console.ReadKey();
        }
    }
}
```

##### **آیا می‌توانید ثابت کنید که متد مخرب به طور ضمنی توسط Garbage Collector فراخوانی می‌شود؟**

بله. لطفاً به مثال زیر نگاهی بیندازید. در مثال زیر، درون متد Main، یک نمونه از کلاس DestructorDemo ایجاد کرده‌ایم و سپس شیء را null قرار داده و متدهای GC.Collect را سه بار فراخوانی می‌کنیم. نکته‌ای که باید به آن توجه کنید این است که متد destructor یا finalizer چند بار و چه زمانی اجرا می‌شود.

```csharp
using System;

namespace DestructorExample
{
    class DestructorDemo
    {
        public DestructorDemo()
        {
            Console.WriteLine("DestructorDemo Object Created");
        }
        ~DestructorDemo()
        {
            Console.WriteLine("DestructorDemo Object Destroyed");
        }
    }

    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("Main Method Execution Started");
            DestructorDemo obj1 = new DestructorDemo();
            
            //Making obj1 ready for Garbage Collection
            obj1 = null;
            GC.Collect();
            Console.WriteLine("Some Statement Executed Inside Main Method");

            obj1 = null;
            GC.Collect();
            Console.WriteLine("Some More Statement Executed Inside Main Method");

            GC.Collect();
            Console.WriteLine("Main Method Execution End");
            Console.ReadKey();
        }
    }
}
```

همانطور که می‌بینید، تابع مخرب (Destructor) به طور ضمنی متد Finalize را در کلاس پایه شیء فراخوانی می‌کند. بنابراین، فراخوانی تابع مخرب به طور ضمنی به صورت زیر ترجمه می‌شود:

![نحوه پیاده‌سازی مخرب‌ها در سی‌شارپ](https://dotnettutorials.net/wp-content/uploads/2018/07/word-image-1131-9.png "نحوه پیاده‌سازی مخرب‌ها در سی‌شارپ")

همانطور که در کد بالا مشاهده می‌کنید، تابع مخرب به متد Finalize تبدیل می‌شود. بنابراین، هر کدی که درون تابع مخرب نوشته باشید، آن کدها درون بلوک try متد Finalize قرار می‌گیرند و کد آن بلوک اجرا شده و منابع را پاکسازی می‌کند. به محض اینکه بلوک try اجرای خود را کامل کرد، بلوک finally اجرا خواهد شد. finally یک بلوک است که اجرای آن تضمین شده است. از بلوک finally، ما دوباره متد finally کلاس والد را فراخوانی می‌کنیم و متد finalize کلاس والد نیز دوباره همین کار را انجام خواهد داد.

این طراحی به این معنی است که متد Finalize به صورت بازگشتی برای تمام نمونه‌های موجود در زنجیره وراثت، از مشتق‌شده‌ترین تا کم‌مشتق‌ترین، فراخوانی می‌شود. اجازه دهید این موضوع را با یک مثال درک کنیم.

در مثال زیر، ما سه کلاس ایجاد کرده‌ایم که زنجیره‌ای از وراثت چند سطحی را تشکیل می‌دهند. کلاس First کلاس پایه است، کلاس Second از First مشتق شده است و کلاس Third از کلاس Second مشتق شده است. و از همه مهمتر، هر سه این کلاس‌ها دارای مخرب هستند. در داخل متد Main، یک نمونه از مشتق‌شده‌ترین کلاس یعنی Third ایجاد می‌شود.

```csharp
using System;

namespace DestructorExample
{
    public class First
    {
        ~First()
        {
            Console.WriteLine("Destructor of First Called");
        }
    }

    public class Second : First
    {
        ~Second()
        {
            Console.WriteLine("Destructor of Second Called");
        }
    }

    public class Third : Second
    {
        ~Third()
        {
            Console.WriteLine("Destructor of Third Called");
        }
    }

    class Program
    {
        static void Main(string[] args)
        {
            Third obj = new Third();
            obj = null;
            GC.Collect();
            Console.ReadKey();
        }
    }
}
```

**نکته ۱:** نباید از مخرب‌های خالی در برنامه خود استفاده کنیم. وقتی یک کلاس حاوی یک مخرب است، یک ورودی برای آن مخرب در صف Finalize ایجاد می‌شود. این صفی است که توسط Garbage Collector پردازش می‌شود. وقتی Garbage Collector صف Finalize را پردازش می‌کند، هر مخرب را فراخوانی می‌کند. بنابراین، مخرب‌های غیرضروری، از جمله مخرب‌های خالی، مخرب‌هایی که فقط مخرب کلاس پایه را فراخوانی می‌کنند، یا مخرب‌هایی که فقط متدهای انتشار شرطی را فراخوانی می‌کنند، باعث از دست رفتن بی‌مورد عملکرد می‌شوند.

**نکته ۲:** به عنوان یک توسعه‌دهنده، ما هیچ کنترلی بر زمان فراخوانی و اجرای تابع مخرب نداریم. این جمع‌کننده زباله در چارچوب دات‌نت است که تصمیم می‌گیرد چه زمانی تابع مخرب را فراخوانی کند. جمع‌کننده زباله به صورت دوره‌ای اشیاء را که دیگر توسط برنامه استفاده نمی‌شوند، بررسی می‌کند. اگر شیء‌ای را واجد شرایط تخریب بداند، تابع مخرب (در صورت وجود) را فراخوانی می‌کند و حافظه مورد استفاده برای ذخیره شیء را بازپس می‌گیرد. همچنین می‌توان با فراخوانی **متد GC.Collect**، جمع‌آوری زباله را اجباری کرد، اما اغلب اوقات، باید از این فراخوانی اجتناب کرد زیرا ممکن است مشکلات عملکردی ایجاد کند. 

##### **چه زمانی از Destructor در سی شارپ استفاده کنیم؟**

ممکن است یک سوال در ذهنتان باشد که اگر مدیریت حافظه به طور خودکار توسط garbage collector مدیریت می‌شود، پس چه زمانی باید از Destructor استفاده کنیم؟ به طور کلی، به عنوان توسعه‌دهندگان C#.NET، نیازی نیست نگران مدیریت حافظه باشیم. دلیل این امر این است که garbage collector.NET به طور ضمنی تخصیص و آزادسازی حافظه را برای اشیاء ما مدیریت می‌کند.

با این حال، وقتی برنامه ما با منابع مدیریت نشده مانند پنجره‌ها، فایل‌ها و اتصالات شبکه کار می‌کند، باید از یک مخرب برای آزاد کردن حافظه برای آن منابع مدیریت نشده استفاده کنیم. وقتی شیء واجد شرایط تخریب باشد، جمع‌کننده زباله متد Finalize شیء را اجرا می‌کند. برای درک بهتر، لطفاً به مثال زیر نگاهی بیندازید.

```csharp
using System;

namespace FileHandlinDemo
{
    public class UmmanagedResource
    {
        public UmmanagedResource()
        {
            //Creating Unmanaged Resources
            Console.WriteLine("Unmanaged Resource Created");
        }
        ~UmmanagedResource()
        {
            //Clean up Unmanaged Resources
            Console.WriteLine("Unmanaged Resource Destroyed");
        }
    }
    class Program
    {
        static void Main()
        {
            UmmanagedResource resource = new UmmanagedResource();
            Console.WriteLine("Using Unmanaged Resource");
            resource = null;
            GC.Collect();
            
            Console.ReadKey();
        }
    }
}
```

##### **انتشار صریح منابع با استفاده از الگوی Dispose در سی شارپ:**

اگر برنامه ما از یک منبع خارجی گران‌قیمت استفاده می‌کند، توصیه می‌شود قبل از اینکه garbage collector شیء را آزاد کند، یک روش صریح برای آزادسازی منابع ارائه دهید. برای آزادسازی منبع، توصیه می‌شود یک متد Dispose از رابط IDisposable پیاده‌سازی کنید که پاکسازی لازم را برای شیء انجام دهد. بنابراین، اساساً، ما به کلاسی نیاز داریم که از رابط IDisposable ارث‌بری کند و پیاده‌سازی متدهای Dispose را به شرح زیر ارائه دهد. بنابراین، در اینجا، لطفاً گزینه Implement Interface with Dispose Pattern را همانطور که در تصویر زیر نشان داده شده است، انتخاب کنید.

##### **درک الگوی دفع (Dispose) در سی شارپ:**

1. **disposedValue:** متغیر disposedValue برای شناسایی فراخوانی اضافی. فرض کنید درخواست نابودی منابع را داده‌ایم و اگر دوباره درخواست نابودی شیء را داشته باشیم، این فیلد را بررسی می‌کند و تشخیص می‌دهد که آیا این یک فراخوانی اضافی است یا خیر، و بر اساس این، فعالیت‌های پاکسازی را انجام می‌دهد.
2. **Dispose(bool disping):** ما باید منطق واقعی برای از بین بردن منابع مدیریت‌شده و مدیریت‌نشده را درون این متد بنویسیم. مقدار متغیر dispose که این متد به عنوان پارامتر ورودی می‌گیرد، مشخص می‌کند که آیا نیاز به از بین بردن منابع مدیریت‌شده داریم یا خیر. علاوه بر این، مقدار متغیر disposedValue را بررسی می‌کند تا بررسی کند که آیا نیاز به انجام فعالیت پاکسازی دارد یا خیر. اگر درخواست برای اولین بار ارسال شود، مقدار متغیر disposedValue برابر با false خواهد بود و در این صورت، فعالیت پاکسازی را انجام می‌دهد و در پایان، مقدار را true قرار می‌دهد. بنابراین، وقتی یک درخواست تکراری می‌آید، فعالیت پاکسازی را انجام نمی‌دهد.
3. **~UmmanagedResource():** به طور پیش‌فرض، خواهید دید که کد مخرب یا نهایی‌ساز به صورت کامنت درآمده است. اما اگر می‌خواهید منابع را از طریق مخرب از بین ببرید، باید این مخرب یا نهایی‌ساز را از حالت کامنت خارج کنید. اما، به عنوان بخشی از این مخرب، باید کد پاکسازی را برای منابع مدیریت‌شده بنویسید و سپس باید متد Dispose(false) را برای پاکسازی منابع مدیریت‌نشده فراخوانی کنید. شما فقط در صورتی باید یک نهایی‌ساز یا مخرب را بازنویسی کنید که متد Dispose(bool disposing) کدی برای آزادسازی منابع مدیریت‌نشده داشته باشد.
4. **Dispose():** این متدی است که باید از کد خود برای پاکسازی منابع مدیریت‌شده و مدیریت‌نشده فراخوانی کنیم. به عنوان بخشی از این متد، باید **متد Dispose(true)** را فراخوانی کنیم که در واقع قرار است منابع مدیریت‌شده و مدیریت‌نشده را پاکسازی کند. باز هم، اگر متد destructor یا finalizer را override کنید، باید **متد GC.SuppressFinalize(this)** را فراخوانی کنید که به garbage collector می‌گوید که destructor را نادیده بگیرد، یعنی به CLR (Garbage Collector) اطلاع می‌دهد که destructor یا finalizer را فراخوانی نکند. فقط در صورتی که destructor یا finalizer را override کرده باشید، باید این متد را از حالت کامنت خارج کنید.

الگوی دفع فوق می‌تواند عملکرد برنامه را به میزان قابل توجهی بهبود بخشد. حتی با وجود این کنترل صریح بر منابع، تابع نهایی‌ساز یا مخرب به عنوان یک محافظ برای پاکسازی منابع در صورت عدم موفقیت فراخوانی متد دفع عمل می‌کند. بیایید چند دستور چاپ بنویسیم و خروجی را ببینیم. کد کامل مثال در زیر آمده است.

```csharp
using System;

namespace DestructorDemo
{
    public class UmmanagedResource : IDisposable
    {
        #region IDisposable Support
        private bool disposedValue = false; // To detect redundant calls

        protected virtual void Dispose(bool disposing)
        {
            if (!disposedValue)
            {
                if (disposing)
                {
                    //Write Code Here to Destroy the Managed Resources
                    Console.WriteLine("Managed Resources Destroyed by Dispose Method");
                }

                //Write Code Here to Destroy the Umanaged Resources
                Console.WriteLine("Unmanaged Resources Destroyed by Dispose Method");
                disposedValue = true;
            }
            else
            {
                Console.WriteLine("Resources are Already Destroyed by Dispose Method");
            }
        }

        ~UmmanagedResource()
        {
            //Write Code here to Destroy the Object
            Console.WriteLine("Request Comes to Destructor to Destroy the Resources");
            Dispose(false);
        }

        // This code added to correctly implement the disposable pattern.
        public void Dispose()
        {
            //Write Code here to Destroy the Object
            Console.WriteLine("Request Comes to Dispose Method to Destroy the Resources");
            Dispose(true);
            GC.SuppressFinalize(this);
        }
        #endregion
    }
    class Program
    {
        static void Main()
        {
            UmmanagedResource resource = null;
            try
            {
                resource = new UmmanagedResource();
                Console.WriteLine("Using Resources");
            }
            finally
            {
                if (resource != null)
                {
                    Console.WriteLine("Calling Dispose Method to Destroy Resources");
                    resource.Dispose();
                }
            }

            //Trying to Call the Dispose Method again
            Console.WriteLine();
            Console.WriteLine("Trying to Call the Dispose Method Again To Destroy Resources");
            resource.Dispose();
            Console.ReadKey();
        }
    }
}
```
